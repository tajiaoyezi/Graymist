# web-ui Specification

## Purpose

平台前端的设计系统与页面外观契约 —— 设计令牌与主题(light/dark/accent)、应用外壳(深色侧栏 + 顶栏 + 主区)、模型仓库与部署管控台的视觉呈现、推理 Playground(由 a3 接通推理 API)与监控仪表盘(由 a4 接通监控 API)的页面呈现。由 `ui-redesign` 引入(视觉框架 + 主题切换 + 4 页),严守 v1.0 范围(不含超纲页面/功能件);只改前端呈现,不改变 `model-registry`/`endpoint-deployment` 的功能行为。推理 Playground 已由 a3 接通真实推理 API(`inference-api`);监控仪表盘已由 a4 接通真实监控 API(`monitoring`)。v1.0 四页至此全部接通真实后端。

## Requirements

### Requirement: 设计系统与主题令牌
平台前端 SHALL 以一套语义化设计令牌(CSS 变量)统一呈现:主色 `--accent` 与其派生柔色 `--accent-soft`、背景 `--bg`、面板 `--panel`、表面 `--surface`/`--surface2`、边框 `--border`/`--border-soft`、文字 `--text`/`--text2`/`--muted`/`--faint`/`--faint2`、侧栏 `--sidebar`/`--sidebar-line`;正文字体 Manrope、代码/数字字体 JetBrains Mono(中文走系统字体 PingFang SC / Microsoft YaHei 回退,与原型一致)。令牌 MUST 提供浅色与深色两套取值,组件 MUST 通过语义令牌取色而非写死色值;`--accent-soft` MUST 由 `--accent` 派生(`color-mix`),使主题色切换时强调底色联动。

#### Scenario: 浅色为默认主题
- **WHEN** 用户首次打开应用且无历史偏好
- **THEN** 根元素处于浅色主题(`--bg:#f1f5f9`、`--panel:#ffffff`、`--accent:#4f46e5`),正文以 Manrope 渲染,中文以系统字体渲染

#### Scenario: 令牌驱动呈现
- **WHEN** 任一页面渲染卡片/表格/徽章/激活态
- **THEN** 其颜色取自语义令牌(panel/border/text/muted/accent-soft 等),数字与代码块使用 JetBrains Mono(`.mono`),无写死色值

### Requirement: 主题与主题色切换
平台 SHALL 提供浅色/深色切换与主题色(6 色)切换,纯前端实现,选择 MUST 持久化到 localStorage 并在下次进入时恢复;切换 MUST 立即作用于全局且不联后端。初始主题 MUST 在首屏渲染前确定以避免闪烁。

#### Scenario: 切换深色并持久化
- **WHEN** 用户点击"深色"
- **THEN** 根元素切换到深色令牌(`--bg:#0a0f1a`、`--panel:#151c2b`、`--sidebar:#0c121f` 等),刷新页面后仍为深色

#### Scenario: 切换主题色联动强调底色
- **WHEN** 用户选择某个主题色(如 `#0d9488`)
- **THEN** `--accent` 即时更新,强调元素(主按钮、激活态、`--accent-soft` 徽章底色)随之变色,且该选择被持久化

### Requirement: 应用外壳与导航
平台 SHALL 以「深色侧栏(248px)+ 顶栏 + 主内容区」外壳承载所有页面。侧栏顶部展示品牌标识与分组导航,底部含主题切换;顶栏展示当前页面标题。侧栏导航 MUST 且仅 MUST 列出 v1.0 的四个页面:模型仓库、部署管控台、推理 Playground、监控仪表盘;「创建模型」MUST NOT 作为导航项(其入口在模型仓库页内)。当前页对应的导航项 MUST 高亮。

#### Scenario: 导航仅含四页
- **WHEN** 应用加载完成
- **THEN** 侧栏导航恰好包含 模型仓库 / 部署管控台 / 推理 Playground / 监控仪表盘 四项,不出现「创建模型」及网关、成本、告警、审计、团队、设置等条目

#### Scenario: 当前页高亮与标题
- **WHEN** 用户进入"部署管控台"
- **THEN** 侧栏中该项处于激活态,顶栏标题显示"部署管控台"

### Requirement: 模型仓库视觉呈现
模型仓库 SHALL 以设计系统呈现列表、模型详情、版本详情与表单四部分(对应 §2.5):
- **列表**:任务类型筛选 tab、名称搜索框、模型卡片网格(名称/描述/标签/版本数/状态点)、页内「创建模型」按钮入口;
- **模型详情**:顶部「← 返回模型列表」入口;左右双栏——左为版本列表(版本号 + 状态徽章 + 框架 + 时间)与版本间指标对比表,右为基本信息与输入/输出 Schema 代码块;
- **版本详情**:Schema 格式化渲染、性能指标、变更说明、状态流转操作按钮;
- **表单**:创建模型 / 新建版本表单,含 Schema 编辑器(JSON 输入)。

本需求只改呈现,MUST NOT 改变 `model-registry` 的功能行为(状态流转、Schema 校验、筛选搜索等语义保持)。

#### Scenario: 列表卡片、筛选与创建入口
- **WHEN** 用户打开模型仓库
- **THEN** 以卡片网格展示模型,可按任务类型切换 tab、按名称搜索,并能经页内「创建模型」按钮进入创建页;既有筛选/搜索/创建行为不变

#### Scenario: 模型详情双栏、返回与 Schema 展示
- **WHEN** 用户打开某模型详情
- **THEN** 顶部有「← 返回模型列表」入口,左栏显示版本列表与指标对比表、右栏显示基本信息与格式化的输入/输出 Schema 代码块

#### Scenario: 版本详情与表单保留功能语义
- **WHEN** 用户打开版本详情或创建/新建版本表单
- **THEN** 版本详情呈现 Schema 格式化/性能指标/变更说明/状态流转按钮,表单含可编辑的 Schema 编辑器,状态流转与 Schema 校验等功能行为不变

### Requirement: 部署管控台视觉呈现
部署管控台 SHALL 以设计系统呈现端点表格(端点名/URL、状态徽章、模型·版本(A/B)、资源占用、操作按钮),creating 状态 MUST 有进行中的脉冲提示;**创建端点以部署管控台内弹窗(Modal)呈现**(由「创建端点」按钮触发),含选模型、A/B 版本多选与权重控件(滑块 + 同步数值框,实时显示权重之和、非 100% 报错、保留单条越界校验)、资源配额输入与平台剩余配额条(**已用 + 本次待占双段叠加、超额变色**、实时校验)、超时与最大并发。本需求只改呈现,MUST NOT 改变 `endpoint-deployment` 的功能行为。

#### Scenario: 端点表格与状态徽章
- **WHEN** 用户打开部署管控台
- **THEN** 以表格列出端点,状态以带色徽章呈现,creating 端点显示脉冲/进行中提示,操作按钮(启动/停止/重启)随状态可用

#### Scenario: 创建端点弹窗的权重与配额实时反馈
- **WHEN** 用户点「创建端点」打开弹窗并调整 A/B 权重(滑块或数值框)或资源配额
- **THEN** 实时显示权重之和(非 100% 阻止提交并提示)与平台剩余配额双段条(超额时提示部署将被拒绝),滑块与数值框双向同步

### Requirement: 推理 Playground
平台 SHALL 在推理 Playground 页面接通真实推理 API(对应原始需求 §2.7),布局为左右双栏:左为请求面板 —— 选择目标端点(列出可推理的 `running` 端点)、按命中 Model 的 `input_schema` **动态生成输入表单**(按字段渲染对应控件,而非单一 JSON 文本域)、同步/异步模式切换、发送;右为响应面板 —— 展示格式化 JSON 结果与延迟(异步模式显示轮询直至终态),以及本次会话的请求/响应历史区、可回填。调用 MUST 经由 `inference-api`(Schema 校验、并发限流、超时等行为由后端决定并在前端如实呈现)。本页 MUST 严守 v1.0 §2.7 范围,MUST NOT 引入协议切换、SSE 流式、成本估算等超纲元素;所有文案 MUST 走 react-i18next(zh),MUST NOT 硬编码。

#### Scenario: 动态表单按字段生成并同步推理
- **WHEN** 用户选择一个 `running` 端点,页面按命中 Model 的 `input_schema` 各字段生成对应控件(如文本字段→文本框、图片 URL 字段→URL 输入框,而非单一 JSON 文本域),用户填写后以同步模式发送
- **THEN** 页面经推理 API 发起调用,在响应面板展示格式化 JSON 结果与延迟,并把本次请求/响应计入会话历史

#### Scenario: 异步推理轮询至终态
- **WHEN** 用户切换为异步模式并发送
- **THEN** 页面取得任务 ID 后轮询任务状态,直至终态再展示最终结果

#### Scenario: 会话历史回填
- **WHEN** 用户点击会话历史中的某条记录
- **THEN** 该条的输入被回填到请求面板以便再次发送

#### Scenario: 不出现超纲元素
- **WHEN** 浏览推理 Playground 页面
- **THEN** 不存在协议切换、SSE 流式、成本估算等超纲控件,文案均来自 zh 资源表

### Requirement: 监控仪表盘
平台 SHALL 在监控仪表盘页面接通真实监控 API(对应原始需求 §2.8):端点选择、时间范围切换(1 小时 / 24 小时 / 7 天)、自动刷新(开启时按**可配置间隔**轮询,对应原始需求 §2.8『可配置刷新间隔』)、指标卡(覆盖 §2.4 五项:QPS、平均延迟、P99 延迟、错误率、当前并发数),以及 QPS、延迟(均值 + P99)、错误率、A/B 版本对比图与资源总览。图表 MUST 由 `monitoring` API 的真实聚合数据驱动(布局不变,仅换数据源)。本页 MUST 严守 v1.0 §2.8 范围,MUST NOT 引入成本/用量、token/LLM 指标、告警等超纲元素;所有文案 MUST 走 react-i18next(zh)。

#### Scenario: 仪表盘呈现真实聚合数据
- **WHEN** 用户进入监控仪表盘、选择端点与时间范围
- **THEN** 五项指标卡与 QPS/延迟(均值+P99)/错误率/A-B 对比/资源总览图表由 monitoring API 的真实数据渲染

#### Scenario: 切换时间范围重新聚合
- **WHEN** 用户在 1 小时 / 24 小时 / 7 天间切换
- **THEN** 图表按对应分桶(分钟/小时/天)重新拉取并渲染

#### Scenario: 自动刷新轮询(间隔可配置)
- **WHEN** 用户开启自动刷新
- **THEN** 页面按当前所选刷新间隔轮询 monitoring API 刷新指标;关闭后停止轮询

#### Scenario: 调整刷新间隔
- **WHEN** 用户更改刷新间隔(如 5s/10s/30s)
- **THEN** 后续轮询按新间隔执行

#### Scenario: 不出现超纲元素
- **WHEN** 浏览监控仪表盘
- **THEN** 不存在成本/用量、token/LLM 指标、告警等超纲控件,文案均来自 zh 资源表

### Requirement: 范围约束与国际化
本前端改版 MUST 限定在上述四个页面与视觉框架内,MUST NOT 引入原型中的超纲页面(网关/成本/告警/审计/团队/设置)与超纲功能件(角色切换、通知中心、里程碑版本条、EN 语言),亦 MUST NOT 在页面内引入超纲元素(external-api 来源、Playground 协议/SSE 流式/成本估算、监控 token 用量/花费/LLM 指标)。所有界面文案 MUST 走 react-i18next(zh)资源表,MUST NOT 硬编码。

#### Scenario: 不出现超纲入口
- **WHEN** 浏览任一页面与导航
- **THEN** 不存在网关/成本/告警/审计/团队/设置入口,也不存在角色切换、通知中心、里程碑条、EN 切换

#### Scenario: 文案走资源表
- **WHEN** 新增页面渲染文案
- **THEN** 文案来自 zh 资源表的 key,而非硬编码字符串
