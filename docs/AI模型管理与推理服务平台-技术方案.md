# 第一部分:核心能力(v1.0)—— 严格对齐原始需求

> 本部分 = 原需求 8 节功能(2.1–2.8)+「三、特别提醒」6 条约束的技术落地,**逐条对应原文,忠实保留**。唯一的**口径附加**是 §4.2(不新增功能,标【补完·澄清】);§7 技术栈为本项目**已定选型**(实现层面,不构成新增功能、不改变原需求范围);§8 架构留缝为**可选的实现层面建议**(同样不改变原需求范围,v1.0 可不采用)。

## 1. 系统定位与角色

平台是一个**通用模型服务平台(model serving platform)**,核心是四个一等概念的生命周期闭环,以及围绕它的前端控制台。主链路:

```
模型注册 → 版本管理 → 版本就绪(ready) → 端点部署 → A/B 路由 → 同步/异步推理 → 日志记录 → 指标聚合 → 前端管理与监控
```

| 角色 | 职责 |
| --- | --- |
| 算法工程师 | 上传模型、发布版本、维护输入输出 Schema、查看版本指标 |
| 平台工程师 | 管理推理端点、配置资源、配置 A/B 流量策略、启停端点 |
| 业务调用方 | 通过统一 API 调用模型推理(同步 / 异步) |

> v1.0 保持概念清晰,**暂不强制权限边界**(权限/团队边界见已定档 v1.5 / E7)。**v1.0 定位为"自托管模型服务";定位升级见第二部分 §10。**

## 2. 核心领域模型(四个一等概念)

- **Model(模型)** 是逻辑实体(如"文本分类器"),本身不可被调用。
- **Version(版本)** 隶属某个 Model,一个 Model 多个 Version,每个对应一次训练产物,承载"可部署制品"。
- **Endpoint(端点)** 是独立调用入口,有唯一 URL 路径;**一个 Endpoint 可挂载同一 Model 下多个 Version**(A/B),按权重分流。
- **Inference Task(推理任务)** 是一次具体推理调用;同步即时,异步产生可查询任务实体。

关系:**Model 1—N Version,Endpoint N—N Version(经 A/B 绑定带权重),Inference Task 命中某 Endpoint、路由到某具体 Version。**

### 实体字段(v1.0)

**Model**:`id` / `name` / `description` / `task_type`(classification/generation/embedding/custom)/ `input_schema` / `output_schema` / `created_at` / `updated_at`

**ModelVersion**:`id` / `model_id` / `version` / `file_path`(**模拟,不需真实上传**)/ `framework`(PyTorch/ONNX/TensorRT)/ `resource_req`(`{cpu,memory,gpu_vram}`)/ `change_note` / `status`(`draft/validating/ready/archived`)/ `metrics`(`{accuracy,latency,throughput}`,测试时写入)/ `created_at`

**Endpoint**:`id` / `name` / `url_path`(唯一)/ `status`(`creating/running/stopped/failed`)/ `replicas` / `resource_quota`(`{cpu,memory,gpu}`)/ `timeout_ms` / `max_concurrency` / `created_at` / `updated_at`

**EndpointVersionBinding(A/B)**:`endpoint_id` / `model_version_id` / `weight`(同端点下所有绑定 **weight 之和必须 = 100**)

**InferenceLog**:`id` / `endpoint_id` / `version_id`(**实际命中版本**)/ `mode`(sync/async)/ `input_summary` / `output_summary` / `latency_ms` / `status`(成功/超时/错误/429)/ `created_at`

**AsyncInferenceTask**:`id` / `endpoint_id` / `status`(`queued/running/succeeded/failed`)/ `input` / `result` / `created_at` / `finished_at`

**PlatformQuota(平台总配额,可配置)**:`{total_cpu, total_memory, total_gpu}`,用于部署资源预算检查。

## 3. 两个独立状态机

### 版本状态机
```
draft ──→ validating ──→ ready ──→ archived
```
- 只有 **`ready`** 可被部署到端点。
- `validating` 在 v1.0 是**轻量过渡态**(原需求未强制其判定逻辑);**把它"做实"成评测流水线 = 已定档扩展 v1.1.3**。

### 端点状态机
```
creating ──→ running ──→ stopped
     │            │
     └────────────┴──→ failed
```
- 支持 **启动 / 停止 / 重启**;部署/更新/停止操作**异步**(见 4.2)。

> 两个状态机不共享状态字段、不互相触发,分别建模(原需求特别提醒第二条)。

## 4. 后端能力

### 4.1 模型注册与版本管理 `[原 2.1]`
- **Model CRUD**:创建时指定名称、描述、任务类型、输入输出 Schema。
- **版本管理**:上传新版本指定 文件路径(模拟)、框架、运行时资源需求、变更说明;状态流转 `draft→validating→ready→archived`,只有 ready 可部署;**版本间对比**(同 Model 下多 Version 的 准确率/延迟/吞吐,测试时写入)。
- **Schema 驱动**:Model 输入输出 Schema 定义推理请求/响应格式,Playground 与推理 API 均依赖它。

### 4.2 推理端点部署 `[原 2.2]`(后端核心难点)
- **独立端点**:每个 Endpoint 唯一 URL 路径。
- **部署配置**:选择模型版本、副本数、资源配额(CPU/内存/GPU)、超时时间、最大并发数。
- **端点状态流转**:`creating→running→stopped` 及 `failed`;支持启动/停止/重启。
- **A/B 流量分配**:一个端点可同时部署多个版本,按权重百分比分流,**权重之和必须为 100%**。
- **异步部署**:部署/更新/停止 **API 立即返回**,后台执行(**模拟耗时 3~10 秒**),客户端轮询或回调获取最终状态。
- **资源预算检查**:部署时校验请求资源是否超出平台总配额(可配置),超出则**拒绝部署**。
- **资源配额校验口径** **【补完·澄清】**(澄清上述检查的算法,**不新增功能**):
  - 端点资源占用 = `副本数 × 单副本 resource_quota`,按 CPU/内存/GPU **三维分别计算**;
  - 剩余配额 = 平台总配额(`PlatformQuota`)− 所有 `creating/running` 端点占用之和(**累计校验**);
  - 任一维度超出剩余则拒绝部署;**此校验对所有 ready 版本端点的部署普遍适用**(v1.0 无 source 概念,不区分来源)。
  - (来源差异化的配额处理见 §13 v1.1.2 的 1.1.2-a,不在 v1.0。)

### 4.3 推理调用 API `[原 2.3]`
- **同步推理**:请求到端点 → 按 A/B 权重选版本 → 执行推理(**模拟耗时 100ms~3s**)→ 返回结果;超时返回错误。
- **异步推理**:提交任务 → 返回任务 ID → 后台排队执行 → 凭任务 ID 查询结果。
- **Schema 校验**:推理请求需校验输入是否符合 Model 的 Schema。
- **限流(必做,非可选)**:每端点最大并发数限制,超出时——同步返回 **429**,异步**排队等待**。
- **推理日志**:记录端点、版本、输入摘要、输出摘要、延迟、状态。
- **A/B 命中记录**:记录实际命中版本,供后续分析。

### 4.4 监控指标 `[原 2.4]`
- **采集指标**:QPS、平均延迟、P99 延迟、错误率、当前并发数。
- **按时间窗口聚合**:最近 1 小时按分钟、最近 24 小时按小时、最近 7 天按天。
- **按版本分组**:支撑 A/B 效果对比。
- **查询 API**:供前端图表直接消费。

## 5. 前端能力

### 5.1 模型仓库页面 `[原 2.5]`
- 模型列表(卡片/表格,按任务类型筛选、搜索);模型详情(基本信息 + 版本列表 + 版本间指标对比);版本详情(Schema 格式化渲染、性能指标、变更说明、状态流转操作按钮);创建模型/新建版本表单(含 Schema 编辑器)。

### 5.2 部署管控台 `[原 2.6]`
- 端点列表(状态、关联模型版本、资源占用);创建/编辑端点(选模型 → 多选版本用于 A/B → 配置权重,**实时显示权重之和、非 100% 报错**;资源配额配置,**实时显示平台剩余配额**);状态实时更新(部署中加载态,完成自动刷新);操作按钮(启动/停止/重启,**危险操作二次确认**)。

### 5.3 推理 Playground `[原 2.7]`
- 选择端点 → 按 Model Schema **动态生成输入表单**;发送请求展示结果(格式化 JSON + 延迟);同步/异步切换(异步显示轮询和最终结果);本次会话内请求/响应历史,可回填。

### 5.4 监控仪表盘 `[原 2.8]`
- 选端点和时间范围,展示 QPS 折线、延迟分布(平均 + P99)、错误率折线;A/B 对比视图(按版本分系列);资源总览(CPU/内存/GPU 已用/总量);可配置自动刷新。

## 6. 关键设计约束(不可削弱,= 原需求特别提醒 6 条)

1. **领域理解是前提** — 四个核心概念关系先理清(§2)。
2. **两个状态机独立** — 版本机与端点机分别建模(§3)。
3. **A/B 权重一致性(原子性)** — 改权重必须原子,**绝不出现权重之和 ≠ 100% 的中间态**。
4. **异步操作的反馈** — 部署与异步推理都异步,前端需合理轮询/加载态。
5. **并发限流是必须项** — 限流必做,非优化项。
6. **Schema 是一等公民** — 输入输出 Schema 贯穿全系统(版本定义 → 推理校验 → Playground 表单生成)。

## 8. v1.0 架构决策点(现在便宜,将来昂贵 —— 留缝)

> 以下四条均为**可选的实现建议**(成本极低的前瞻性留缝),**不属于原需求、v1.0 可不采用**;采用与否不影响 v1.0 功能完整性,只影响日后扩展的接入成本。

1. **状态流转留一条"变更日志"的缝**:两个状态机的每次流转、A/B 权重每次变更,除更新当前态外**向一张 append-only 变更日志追加一条不可变记录**(操作、前后值、时间、操作人 —— 其中 `操作人` 在 v1.5/E7 建立管理侧真实身份前记为占位标识 `local-admin`/`system`,届时起填真实用户,历史不回填,见 D12)。这只是**普通 CRUD 表旁边的一张只追加日志表**,零架构承诺、成本极低,却让扩展期的审计/配置历史/回滚能直接长上去。**审计(E9,已定档 v1.3)即直接复用这条缝。**(A/B 权重原子性由约束 6.3 的事务写入保证,与本缝无关;本缝只负责留历史。是否进一步升级为完整事件溯源,见 §23,**不必现在决定**。)
2. **A/B 权重数据模型不要写死成"只能静态权重"**:把"端点当前挂载哪些版本、各自多少权重"建模成**可整体替换的配置**。→ 渐进发布(E2)能直接接上。
3. **端点推理抽象留一个流式出口**:v1.0 只做 sync/async 返回完整结果,但端点 → 版本选择 → 执行链路**预留分块产出(streaming)出口**。→ **已在 v1.2 实现 SSE 流式**,改动局限在执行层。
4. **前端文案与本地化留一条 i18n 缝**:前端**不硬编码界面文案**,全部走文案资源表(key→文案,选定框架如 react-i18next);时间统一以 **UTC 存储**、展示层按 locale 格式化(日期/数字/货币/时区)。v1.0 仍是单语言、界面无变化,只是把"将来加语言"的成本从"满屏抠硬编码"降到"补一份翻译表"。→ **完整多语言已定档 v1.5**(见 §18 1.5-d / D14)。

## 9. 贯穿性架构原则(适用所有扩展版本)

| 编号 | 原则 | 说明 |
| --- | --- | --- |
| **P1** | 平台不自造推理引擎 | 平台只**编排/调用**现成 runtime(onnxruntime 库 / Ollama / vLLM / LMDeploy / 公网 API),绝不重写推理引擎 |
| **P2** | 执行层 = canonical 内核 + 协议适配器(N+M) | 内部用统一 canonical 表示;南向(调上游)/北向(对外暴露)复用同一批适配器;Schema 校验、日志、指标、A/B 全在 canonical 层 |
| **P3** | 模型来源维度 `source` | `source ∈ {external-api, local-engine, uploaded-file}`,**归属判定 = 平台是否管理该后端生命周期**(只调 HTTP API → external-api;平台拉起/管理实例 → local-engine;平台进程内运行 → uploaded-file);`Version` 按 source 分叉字段(见 §11 v1.1 / §15 v1.2 / §19 v2.0) |
| **P4** | 两个模型家族 | **通用 ML**(ONNX,张量 I/O,分类/嵌入) vs **LLM**(chat/token,生成);I/O、Schema、指标(准确率 vs TTFT/tokens-s)不同,适配层分别处理 |

## 10. 定位升级 【超纲·产品决策】

原需求隐含定位是「自托管模型服务平台」。本规划主动升级为:

> **统一模型管理 + 评测 + 网关平台** —— 可纳管三类来源,用同一套「注册 → 部署 → 路由 → 推理 → 监控」统一治理:
> 1. **公网 / 远程 API 模型**(OpenAI / Anthropic / 通义 / DeepSeek …)→ `source=external-api`(见 §11 1.1-a)
> 2. **本地引擎托管模型**(Ollama / vLLM / LMDeploy):
>    - **Ollama**:平台只调它的 HTTP API(`base_url` 指 localhost),机制同公网 API → 技术上归 `source=external-api`(§11 1.1-a);
>    - **vLLM / LMDeploy**:平台自管实例生命周期 → `source=local-engine`(§15)。
> 3. **平台自运行的上传模型**(ONNX,内嵌 onnxruntime)→ `source=uploaded-file`(§19,v2.0)。
>
> 注:本节是按**部署拓扑**做的**产品分类**,与技术字段 `source` 不一一对应——关键看 **平台是否管理该后端生命周期**(P3)。
>
> **版本线边界**:**v1.x 平台始终是"网关/编排者"**(推理交给外部引擎,符合 P1);**v2.0 平台跨入"自运行引擎宿主"**(内嵌 onnxruntime,平台进程内跑模型)。这是平台性质的一次重大跨越,故用 major 版本号。

## 14. v1.1.3 — validating 做实(评测 + 硬门禁,= 原 E6)

- **触发**:版本进入 `validating` 时,对**注册的测试集**跑评测,算出 准确率/延迟/吞吐,写入 `metrics`(测试集来源 = 创建版本时附带引用,沿用 `file_path` 的 mock 思路,可模拟)。
- **门禁 = 硬门禁(D1)**:产出 `validation_result: pending/passed/failed`(**独立字段,与 `status` 生命周期解耦**);
  - 仅 `validation_result = passed` 可晋升 `ready`;
  - 不达标 → `validation_result = failed`,**版本停在 `validating`、永远进不了 `ready`** → 因"只有 ready 可部署"被**自动挡住**(硬门禁靠"进不了 ready"实现);
  - **不新增 `failed` 版本状态**(原版本状态机 `draft→validating→ready→archived` 无 failed,**不改动**,符合约束 6.2);
  - 重试 = 上传新版本(失败版本可 `archived`)。
- **可选子项**:Schema 一致性校验(校验模型实际 I/O 是否符合声明的 Model Schema)。
- **依赖**:真实执行(v1.1 起 external 即满足;local-engine 走 v1.2、uploaded 走 v2.0,届时回溯生效)。

## 23. 关于事件溯源(Event Sourcing)—— 一个可选的进阶方向,**不必现在拍**

v1.0 真正要做的,只是在 §8.1 留一条 **append-only 变更日志缝**(零承诺、成本极低):每次状态流转 / 配置变更顺手追加一条不可变记录。审计(E9,v1.3)、配置历史(E3)、回滚(E2)都能直接长在这条缝上,而且**任何时候都能加,不需要重写持久层**。

**是否进一步升级为"完整事件溯源"**(让事件日志当唯一真相、当前态是它的投影,从而获得时间旅行、把 `DeploymentRevision`/`TrafficChangeLog`/`RollbackRecord`/`AuditLog` 塌缩成一条事件日志 + 投影)——**是个可选的进阶方向,不必在 v1.0 决定**,等真要做审计/回滚/时间旅行时再评估即可。它是有真复杂度的承诺(投影、回放、最终一致性),不是免费午餐。

> 澄清:**A/B 权重原子性(约束 6.3)由事务写入保证,不依赖事件溯源**;变更日志缝只负责留下历史。

## 25. 未定档扩展详细说明

- **E2 渐进式发布 + 自动回滚【高杠杆】**:静态 A/B 升级为自动 rollout(5%→25%→50%→100%,按时间/指标推进),护栏指标越线自动回滚。依赖 E3 / §8.2 / 变更日志缝 / **v1.4 告警信号**。
- **E3 部署修订与配置历史【增强·地基】**:每次端点配置变化生成不可变修订(含 `rollback_from_revision_id`),一键回滚/审计/发布历史的地基,可吸收 `TrafficChangeLog`/`RollbackRecord`。本质 = §8.1 变更日志缝在端点配置上的投影/视图。
- **E4 影子部署【增强】**:复制线上流量打到候选版本,只离线比对、绝不返回给用户。上 A/B 前最安全的真实流量验证。
- **E5 自动扩缩容【增强】**:按 QPS/并发自动扩缩副本数,配合资源配额。依赖 **v1.4 告警信号**;**仅对 local-engine(v1.2)/ uploaded(v2.0)后端有意义**(external API 由供应商扩缩)。
- **E12 分布式追踪【增强】**:推理日志升级为全链路 trace(端点 → 版本选择 → 执行 → 响应)。
- **E13 异步任务增强【增强】**:状态扩展 `queued/running/succeeded/failed/canceled/timeout`,加取消/重试/回调/幂等键/队列位置/结果过期。接口:`POST /inference/jobs`、`GET /inference/jobs/{id}`、`.../cancel`、`.../retry`。
- **E14 开发者中心 / OpenAPI / SDK / 错误码【增强】**:面向业务方的接入门户,文档/示例可从 Schema 生成。**= 对外服务层(§20)的后置部分。**
- **E16 Model Card【增强】**:模型从"技术资产"升级为"可被业务理解的服务资产"(用途/不适用场景/输入要求/输出解释/负责人/当前线上版本…)。
- **E17 推理编排 / DAG【靠后·野心款】**:多模型串成 DAG(预处理 → embedding → 检索 → 分类),通向 Agent Runtime 思路,但很重,明确标为 future work。
- **E18 批量推理【增强】**:数据集级离线批跑(提交数据集 → 后台批处理 → 下载结果),与"每请求异步"不同。
- **E19 Prompt / 模板版本管理【靠后 → 可上调】**:把 prompt 当成与模型版本解耦的独立版本化产物,可独立 A/B prompt。原定位"仅当走 LLM 才做";**因本平台已明确走 LLM(vLLM/Claude/流式),其优先级可从'靠后'上调评估。**
