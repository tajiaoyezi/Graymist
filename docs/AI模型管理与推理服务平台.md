# AI 模型管理与推理服务平台

- ***\*后端\****：模型注册与版本管理、推理端点部署、同步/异步推理调用、A/B 流量分配、指标监控；
- ***\*前端\****：模型仓库浏览、部署管控台、推理调试 Playground、监控仪表盘。

系统面向 AI 团队——算法工程师上传模型并发布版本，平台工程师管理推理端点和流量策略，业务方通过统一 API 调用模型推理。

### 注意事项

本项目复杂度较高，涉及模型生命周期管理、部署状态机、流量路由、异步推理队列等后端设计难点，同时要求前端具备实时数据展示和复杂表单交互能力。

## 二、功能需求

### 2.1 模型注册与版本管理（后端）

***\*描述\****：模型仓库的核心，管理模型及其版本的完整生命周期。

***\*要求\****：

- 模型是一个逻辑实体（如"文本分类器"），模型下有多个版本（v1、v2...），每个版本对应一次训练产物。
- 模型 CRUD：创建模型时指定名称、描述、任务类型（分类/生成/嵌入/自定义）、输入输出 Schema。
- 版本管理：
  - 上传新版本时指定：模型文件路径（模拟，不需要真实上传）、框架（PyTorch/ONNX/TensorRT）、运行时资源需求（CPU/内存/GPU 显存）、变更说明。
  - 版本状态流转：`draft` → `validating` → `ready` → `archived`。只有 `ready` 的版本可以被部署。
  - 版本间对比：查询同一模型下多个版本的性能指标（准确率/延迟/吞吐，数据由测试时写入）。
- 模型的输入输出 Schema 定义推理接口的请求/响应格式，后续 Playground 和推理 API 均依赖此 Schema。

### 2.2 推理端点部署（后端）

***\*描述\****：将 ready 状态的模型版本部署为可调用的推理端点。这是后端核心难点。

***\*要求\****：

- 推理端点（Endpoint）是独立的调用入口，每个端点有唯一的 URL 路径。
- 部署配置：选择模型版本、副本数、资源配额（CPU/内存/GPU）、超时时间、最大并发数。
- 端点状态流转：`creating` → `running` → `stopped`，以及 `failed`。支持启动、停止、重启操作。
- ***\*A/B 流量分配\****：一个端点可同时部署多个模型版本，按权重百分比分配流量（如 v2: 80%, v3: 20%）。权重之和必须为 100%。
- 端点的部署/更新/停止操作是异步的——API 立即返回，后台执行（模拟耗时 3~10 秒），客户端通过轮询或回调获取最终状态。
- 资源预算检查：部署时校验请求的资源是否超出平台配额（平台总配额可配置），超出则拒绝部署。

### 2.3 推理调用 API（后端）

***\*描述\****：业务方通过统一 API 调用推理端点，支持同步和异步两种模式。

***\*要求\****：

- ***\*同步推理\****：请求发到端点 → 根据 A/B 权重选择模型版本 → 执行推理（模拟耗时 100ms~3s）→ 返回结果。超时返回错误。
- ***\*异步推理\****：提交推理任务 → 返回任务 ID → 后台排队执行 → 客户端凭任务 ID 查询结果。
- 推理请求需校验输入是否符合模型的 Schema 定义。
- ***\*限流\****：每个端点有最大并发数限制，超出时同步请求返回 429，异步请求排队等待。
- 推理日志：记录每次调用的端点、版本、输入摘要、输出摘要、延迟、状态。
- A/B 路由需记录实际命中的版本，用于后续分析。

### 2.4 监控指标（后端）

***\*描述\****：采集并聚合推理端点的运行指标。

***\*要求\****：

- 采集指标：QPS、平均延迟、P99 延迟、错误率、当前并发数。
- 按时间窗口聚合（最近 1 小时按分钟、最近 24 小时按小时、最近 7 天按天）。
- 按模型版本分组的指标（支撑 A/B 效果对比）。
- 提供查询 API，前端图表直接消费。

### 2.5 模型仓库页面（前端）

***\*描述\****：浏览和管理模型及其版本。

***\*要求\****：

- 模型列表：卡片或表格展示，按任务类型筛选、搜索。
- 模型详情：基本信息 + 版本列表（状态、创建时间、资源需求） + 版本间指标对比（表格或图表）。
- 版本详情：Schema 展示（JSON 格式化渲染）、性能指标、变更说明、状态流转操作按钮。
- 创建模型/新建版本的表单（含 Schema 编辑器——JSON 输入或结构化表单）。

### 2.6 部署管控台（前端）

***\*描述\****：管理推理端点的部署和流量配置。

***\*要求\****：

- 端点列表：展示所有端点及其状态、关联的模型版本、资源占用。
- 创建/编辑端点：
  - 选择模型 → 选择版本（可多选用于 A/B）→ 配置流量权重（滑块或输入框，实时显示权重之和，非 100% 时报错）。
  - 资源配额配置（副本数、CPU、内存、GPU），实时显示平台剩余配额。
- 端点状态实时更新：部署中显示进度/加载态，完成后自动刷新状态。
- 操作按钮：启动/停止/重启，危险操作需二次确认。

### 2.7 推理 Playground（前端）

***\*描述\****：交互式推理调试工具，方便快速测试模型效果。

***\*要求\****：

- 选择端点 → 根据模型 Schema ***\*动态生成\****输入表单（如文本分类模型生成文本输入框，图像模型生成图片 URL 输入框）。
- 发送推理请求，展示返回结果（格式化 JSON + 延迟信息）。
- 支持切换同步/异步模式。异步模式下显示任务状态轮询和最终结果。
- 请求/响应历史记录（本次会话内），可快速回填之前的输入。

### 2.8 监控仪表盘（前端）

***\*描述\****：可视化展示推理端点的运行指标。

***\*要求\****：

- 选择端点和时间范围后，展示：QPS 折线图、延迟分布图（平均 + P99）、错误率折线图。
- A/B 对比视图：同一图表上按版本分系列展示指标，直观对比不同版本表现。
- 资源总览：平台资源配额使用率（CPU/内存/GPU 的已用/总量）。
- 支持自动刷新（可配置刷新间隔）。

## 三、特别提醒

- ***\*领域理解是前提\****：模型/版本/端点/推理任务四个核心概念的关系必须在方案阶段理清，否则后续代码会反复返工。
- ***\*两个状态机\****：版本状态机（draft→ready→archived）和端点状态机（creating→running→stopped/failed）各自独立，不要混淆。
- ***\*A/B 权重一致性\****：修改流量权重时必须保证原子性——不能出现权重之和不等于 100% 的中间态。
- ***\*异步操作的反馈\****：部署和异步推理都是异步的，前端需要合理的轮询/加载态设计，避免用户困惑。
- ***\*并发限流不是可选项\****：推理 API 面向业务方高频调用，限流是必须实现的功能，不是"优化项"。
- ***\*Schema 驱动\****：模型的输入输出 Schema 贯穿整个系统（版本定义→推理校验→Playground 表单生成），是一等公民。
